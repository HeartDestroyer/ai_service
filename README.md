# AI-cервис обогащения и создания

Изолированный, высокопроизводительный сервис, выполняющий роль интеллектуального ядра. Автоматизирует процессы B2B-взаимодействия: интеллектуальное обогащение профилей компаний из разрозненных источников, генерацию аналитических саммари, выявление болей клиентов и автоматизированный аутрич (создание писем)

## 🏗️ Архитектурная концепция: Модульный монолит (Clean Architecture)

Система построена на принципах строгой изоляции слоев и независимости бизнес-контекстов (Vertical Slices).

* **Независимость модулей:** Бизнес-процессы (`enrichment`, `outreach`, `summary`) физически разделены. Каждый модуль инкапсулирует собственные эндпоинты, бизнес-логику, интерфейсы и работу с базой данных. При необходимости любой модуль может быть вынесен в отдельный микросервис без изменения кода
* **Изоляция моделей данных (DTO):** Настроено жесткое разделение структур данных. Публичные Pydantic-схемы (на уровне API) никогда не объединяются с приватными доменными DTO (на уровне бизнес-логики). Это гарантирует, что изменения во внешнем API-контракте не сломают ядро системы
* **Инкапсуляция ORM и паттерн Repository:** Слой бизнес-логики не работает с объектами БД (ORM-моделями). Репозитории модулей преобразуют данные из базы и возвращают наверх исключительно чистые DTO
* **Децентрализованный Unit of Work (UoW):** Базовый UoW и инфраструктурные механизмы транзакций располагаются в `core/infrastructure/uow.py`; конкретные реализации UoW создаются локально внутри бизнес-модулей
* **Shared Kernel (Общее ядро):** Низкоуровневые инструменты без бизнес-логики (например, «глупый» HTTP-клиент для связи с провайдерами ИИ) вынесены в `infrastructure/pkg` и могут безопасно переиспользоваться любыми модулями

## 🧠 Работа с ИИ

* **Слепая бизнес-логика:** Use Cases (сервисы модулей) не знают, какая именно LLM (Gemini, OpenAI или локальная модель) обрабатывает запрос, и не содержат внутри себя текстов промптов
* **Управление контекстом в инфраструктуре:** Системные инструкции, выбор подходящей модели (например, быстрая/дешевая для парсинга, мощная для написания писем) и маппинг ответов реализованы в слое `infrastructure/adapters` каждого модуля

## 🛡️ Безопасность и Инфраструктура

* **Сетевой периметр:** Публикация API выполняется через Traefik, backend-сервисы изолированы в отдельной docker-сети
* **Аутентификация и контроль нагрузки:** Эндпоинты защищены API-ключами; для анонимных клиентов включен rate-limit (по IP, окно и лимит настраиваются в `core/middleware/security.py`) с хранением счетчиков в Redis
* **Анти-спуфинг IP:** для лимитов используется только `request.client.host`, заголовок `X-Forwarded-For` игнорируется
* **Сквозная обработка (Middleware):** Заголовки безопасности, CORS, rate-limit и централизованное логирование вынесены в `core/*`

## 🚀 Локальный запуск

1. Подготовьте окружение:
   - скопируйте `.env.example` в `.env`
   - заполните обязательные переменные (`SECRET_KEY`, `API_KEY`, пароли БД/Redis и тд)

2. Запуск стэка (из корня репозитория, где лежит `.env`):

```bash
docker compose up -d --build
```

Compose читает `.env` из корня для подстановки `${MAX_CONNECT_DB}`, `${DOMAIN}` и т.д. в `deploy/docker-compose.yml`.
`env_file: ../.env` у сервисов по-прежнему передаёт переменные **внутрь** контейнеров — это разные механизмы.

Альтернатива (явный env-файл):

```bash
docker compose --env-file .env -f deploy/docker-compose.yml -f deploy/docker-compose.local.yml up -d --build
```


Это абсолютно правильный архитектурный вопрос. Разделение задач (Task Routing) между разными моделями — это то, что отличает любительский AI-проект (где везде бездумно стоит GPT-4) от enterprise-решения, которое работает быстро, пишет как живой человек и не сжигает бюджет.

В твоем конвейере три совершенно разные по своей природе задачи. Вот как их стоит распределить, если у тебя есть доступ к агрегатору (или напрямую к API разных вендоров).

### Этап 1: Обогащение и поиск данных (Enrichment)

**Задача:** Сходить в поиск/API по ИНН, распарсить сайт компании, вытащить факты, структурировать их и отдать строгий JSON.
**Специфика:** Здесь не нужна креативность. Здесь нужны скорость, дешевизна (запросов будет много), большое контекстное окно (если скармливаешь HTML) и идеальное следование формату JSON.

* **🏆 Основной выбор: `gpt-4o-mini` или `gemini-1.5-flash**`
* *Почему:* Они стоят копейки, работают молниеносно и отлично умеют вызывать функции (Function Calling) для структурирования данных. У Gemini 1.5 Flash огромное контекстное окно (до 1 млн токенов), что идеально, если ты сгружаешь туда сырой текст с корпоративных сайтов.


* **🛠 Настройки:** `temperature: 0.0` или `0.1` (нужна максимальная детерминированность, никаких фантазий).

### Этап 2: Саммари и выявление болей (Summary & Analysis)

**Задача:** Прочитать сырые факты о компании и сделать аналитический вывод. Понять, чем они занимаются, кто их клиенты, и главное — где у них может "болеть" (нехватка кадров, проблемы с логистикой, устаревший софт).
**Специфика:** Требуется сильное логическое мышление (Reasoning) и способность понимать бизнес-контекст.

* **🏆 Основной выбор: `claude-3.5-sonnet` или `gpt-4o**`
* *Почему:* Claude 3.5 Sonnet на данный момент — король аналитики и понимания контекста. Он феноменально вытаскивает неочевидные связи из текста. GPT-4o тоже справится отлично, но иногда делает слишком "шаблонные" выводы.


* **🛠 Настройки:** `temperature: 0.4 - 0.6` (нужен баланс между логикой и способностью генерировать гипотезы).

### Этап 3: Умный аутрич (Генерация письма)

**Задача:** Написать персонализированное письмо, которое "зацепит" ЛПР (лицо, принимающее решения), опираясь на боли из Этапа 2.
**Специфика:** Это самая критичная часть. Если письмо будет звучать как "нейросетевая отписка" (со словами "В современном динамично развивающемся мире...", "Мы предлагаем синергию..."), оно улетит в спам, а конверсия будет нулевой. Нужен высочайший уровень эмпатии и "человечности" текста.

* **🏆 Безоговорочный лидер: `claude-3.5-sonnet` (или `claude-3-opus`)**
* *Почему:* Исторически так сложилось, что модели OpenAI (GPT) пишут очень "пластиково". Их тяжело заставить звучать как настоящий живой менеджер по продажам. Модели от Anthropic (Claude) обладают лучшим слогом на рынке. Они умеют писать кратко, по делу, с правильной долей неформальности и уважения.


* **💡 Альтернатива (Open Source):** Если есть возможность, для этой задачи отлично подходят специализированные LLM, дообученные (Fine-tuned) на реальных успешных холодных письмах (например, на базе `Llama-3`).
* **🛠 Настройки:** `temperature: 0.7 - 0.8` (здесь нужна гибкость языка и креативность).

### Архитектурный совет (Senior-подход)

Так как у тебя модульный монолит, я настоятельно рекомендую в твоем модуле `infrastructure/pkg/polza_ai_client` реализовать паттерн **Fallback (Резервное копирование)**.

Агрегаторы или сами провайдеры иногда "ложатся" (отвечают ошибками 502/429). Твой сервис не должен падать. Логика должна быть такой:

1. Идем за письмом к `claude-3.5-sonnet`.
2. Если таймаут или ошибка провайдера -> ловим `try/except` -> автоматически делаем ретрай (через библиотеку `tenacity`, которая у тебя есть) во вторичную модель, например `gpt-4o`.

Скажи, а как ты планируешь контролировать качество самих писем? Будет ли какой-то этап "Human-in-the-loop" (когда менеджер просматривает письмо перед отправкой и правит его), или система должна отправлять всё полностью на автомате?
